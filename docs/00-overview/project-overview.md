# 项目总览

适用对象：跨组接入方、项目负责人、新维护者  
更新时间：2026-07-01  
关联文档：[项目需求文档](../01-product/product-requirements.md)、[系统架构](../02-architecture/system-architecture.md)

## 1. 项目是什么

实验室管理平台是一个围绕“项目 / 课题组”组织业务的实验室协作系统。它不是单一的耗材系统，也不是单一的文件系统，而是把这些能力挂到统一的项目上下文中：

- 项目成员与角色
- 项目任务与阶段进展
- 耗材申请与审批
- 项目资料与汇报资料
- 会议通知与公告
- AI 问答与知识库

## 2. 当前建设目标

当前仓库既是可演示原型，也是后续跨组整合的主干仓库，建设目标分为两层：

- 产品层：形成一个面向高校实验室的项目制协作平台
- 工程层：形成一个可被不同技术组扩展或重构的主骨架

因此，文档和代码都要明确回答：

- 模块边界在哪里
- 哪些能力已经稳定
- 哪些能力仍处于快速演进
- 外部组接入时应该对接 API、数据库还是模块边界

## 3. 当前实现范围

已落地的核心模块：

- 账号与角色管理
- 项目管理
- 项目成员、任务、结构化汇报
- 项目树与快照
- 耗材管理与审批
- 文件资料与版本记录
- 会议通知与公告
- AI 助手、知识库、FAQ

已落地但仍在持续完善的主线能力：

- 以 `project_id` 为中心的跨模块联动
- 通知与实时刷新
- 项目资料、汇报、项目树的联动展示
- AI 知识库文件导入

## 4. 为什么以本仓库作为整合核心

- 业务主线已经基本闭环，项目模块可连接其他模块
- 工程结构已经模块化，适合作为后续整合骨架
- 数据模型已具备身份、项目、文件、协作和 AI 的基础关联
- 前后端均已形成统一仓库与统一启动方式，便于重构和迁移

## 5. 当前边界

已经明确不在当前仓库首版中深做的内容：

- 真实 NAS SDK 深度接入
- 校内统一认证（CAS / OAuth / SSO）正式适配
- 大规模 E2E 自动化
- 多租户 / 多实验室隔离

这些内容不是禁止实现，而是当前文档和代码都会把它们标为“后续整合主题”，避免误判为已完成能力。
