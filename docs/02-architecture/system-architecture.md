# 系统架构

适用对象：架构师、后端开发、跨组整合方  
更新时间：2026-07-01  
关联文档：[技术栈](./technology-stack.md)、[数据库设计](../03-data/database-design.md)

## 1. 架构风格

当前仓库采用“微内核 + 插件化”结构：

- `packages/core` 提供内核能力
- `apps/api` 负责装配与宿主化
- `plugins/*` 承载具体业务
- `apps/web` 承载前端展示与交互

## 2. 核心层

`packages/core` 提供：

- 认证接口与适配器
- 权限定义
- 审计接口
- 事件总线
- 插件清单与路由注册契约

这些能力应保持“业务无关”，不承载项目、耗材、文件等具体规则。

## 3. 业务插件层

业务插件包括：

- `projects`：项目主轴、任务、成员、项目树、汇报
- `inventory`：耗材、申请、审批、流水
- `files`：文件、版本、项目资料
- `collaboration`：会议、通知、公告
- `ai`：知识库、问答、FAQ
- `hello-world`：示例与健康验证

## 4. API 宿主层

`apps/api` 做三件事：

1. 初始化认证与审计适配器
2. 注册全部插件
3. 暴露统一 HTTP 接口与 SSE 事件流

## 5. Web 层

`apps/web` 当前采用：

- 应用壳层：侧边导航 + 顶栏项目上下文
- 页面分模块组织
- 统一 `useLabData` Hook 负责数据拉取与操作

## 6. 当前最重要的架构原则

- 项目模块是业务主轴
- 插件不直接导入其他插件源码
- 数据库按 schema 分层，不跨插件随意写表
- 对外整合优先依赖接口和文档，不依赖实现细节

## 7. 已知架构欠账

- OpenAPI 与运行时接口仍存在局部不同步
- 部分共享类型仍保留旧模型痕迹
- 个别前端导航角色白名单尚未完全反映 6 级角色体系
